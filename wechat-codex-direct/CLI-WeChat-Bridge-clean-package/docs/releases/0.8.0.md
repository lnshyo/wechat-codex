## 1. Version Check and Auto-Update Capability

### Background

Previously, users had to manually download source code to update the WeChat Bridge. This release introduces a network-based version checking mechanism similar to npm's update workflow, preparing for future npm publishing while maintaining the current installation method.

### Implementation

**Git-Based Version Discovery**
- Uses `git ls-remote --tags origin` to fetch available versions
- Avoids GitHub API rate limits that plagued earlier implementations
- Parses semantic version tags directly from remote repository
- Efficient single-command execution with minimal overhead

**Smart Caching Strategy**
- 24-hour cache stored in `~/.claude/channels/wechat/update-check.json`
- Prevents redundant network requests
- Non-blocking check runs 3 seconds after startup
- Silently fails without affecting bridge operation

**Version Comparison Logic**
- Semantic versioning comparison (major.minor.patch)
- Accurate detection of newer versions
- Distinguishes between equal, greater, and lesser versions

**Manual Check Command**
```bash
wechat-check-update
```

Provides professional output including:
- Current version
- Latest available version
- Clear update instructions
- No emoji for professional presentation

## 2. Media Sending and Attachment Parsing

### Background

Previously, the WeChat Bridge could only send text messages. Users had no way to send images, files, voice messages, or videos through the bridge. This release integrates comprehensive media sending capabilities, merging the standalone `wechat-notify.ts` script into the main transport layer.

### Implementation

**Media Type Support**
- **Images**: PNG, JPG, JPEG, GIF, WebP, BMP (max 20MB)
- **Files**: Any file type (max 50MB)
- **Voice**: MP3, WAV, M4A, OGG, AAC (max 20MB)
- **Videos**: MP4, MOV, MKV, AVI, WebM (max 100MB)

**Attachment Path Resolution**
The bridge now intelligently resolves multiple attachment path formats:

1. **Fenced Code Blocks**
   `````markdown
   ```wechat-attachments
   image C:\Users\user\Desktop\screenshot.png
   file /path/to/document.pdf
   video ~/Videos/recording.mp4
   voice C:\Music\voice_note.m4a
   ```
   ``````

2. **Inline Paths**
   - Windows absolute: `C:\Users\user\file.png`
   - Home-relative: `~/Desktop/file.png`
   - maas-log URLs: `https://server.com/C:\path\to\file.png`

3. **Automatic Type Detection**
   - File extension-based inference
   - Supports both Windows and Unix paths
   - Handles URL-encoded paths

**Dual-Queue Architecture**
Prevents large file uploads from blocking text messages:

- **Text Queue**: High-priority queue for immediate text delivery
- **Attachment Queue**: Separate queue for media uploads
- Both queues operate independently
- Failed attachments don't block subsequent messages

**Final Reply Forwarding Module**
New `bridge-final-reply.ts` module:
- Separates visible text from attachments
- Sends text immediately
- Uploads and sends attachments sequentially
- Graceful error handling per attachment
- Reports failed attachments without crashing

**Media Size Limits**
Configurable via environment variables:

```bash
export WECHAT_MAX_IMAGE_MB=20   # Default: 20MB
export WECHAT_MAX_FILE_MB=50    # Default: 50MB
export WECHAT_MAX_VOICE_MB=20   # Default: 20MB
export WECHAT_MAX_VIDEO_MB=100  # Default: 100MB
```

**Codex Integration**
- Codex final outputs unified as `final_reply` events
- Text and attachments properly separated
- Automatic attachment detection and sending

**MCP Tool Exposure**
Updated MCP tool definitions to expose:
- `wechat_send_image`
- `wechat_send_file`
- `wechat_send_voice`
- `wechat_send_video`

## 3. WeChat Long Poll Failure Prevention

### Background

Previously, any network error during WeChat long polling would crash the entire bridge, requiring manual restart. This was particularly problematic in environments with unstable network connections. This release introduces comprehensive error classification and retry logic.

### Implementation

**Error Classification**
Transport errors are now classified into four categories:

1. **Timeout Errors**
   - Request timeout
   - Connection timeout
   - Header timeout
   - Retryable with exponential backoff

2. **Network Errors**
   - Connection refused
   - Connection reset
   - Host unreachable
   - Network unreachable
   - Retryable with exponential backoff

3. **HTTP Errors**
   - HTTP status codes: 408, 425, 429 (retryable)
   - Other 4xx/5xx codes (may not be retryable)
   - Context-specific handling

4. **Auth Errors**
   - Authentication failures
   - Not retryable (requires user intervention)
   - Exits immediately

**Exponential Backoff Retry**
- Base delay: 1 second
- Max delay: 30 seconds
- Exponential growth: 2^consecutiveFailures
- Capped at 5th power (32 seconds)

**Detailed Error Logging**
New error description system includes:
- Error name and message
- Error code (errno)
- System call (syscall)
- Hostname and address
- Port information
- Cause chain (up to 4 levels deep)

**Non-Recoverable Error Handling**
- Auth failures still exit immediately
- Prevents infinite retry loops
- Clear error messages for user action

```

```
