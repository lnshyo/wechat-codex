**Release Date:** March 24, 2026

Version 0.5.0 represents a transformative release that fundamentally reimagines the bridge architecture. This version introduces native Claude Code support with full dual-terminal parity to Codex, unifies session semantics across all adapters, adds comprehensive cross-platform support for Linux and macOS, and fixes critical state management issues—all totaling 3,595 lines of new code across 25 files.

## 1. Claude Code Companion Integration

### Background

Previously, Claude Code integration was experimental and lacked the polished dual-terminal experience that Codex users enjoyed. This release introduces full native Claude support with a local companion that mirrors the Codex experience exactly.

### Implementation

**Dual-Terminal Architecture**
Claude now supports the same dual-terminal workflow as Codex:

```bash
# Terminal 1: Bridge
wechat-bridge-claude

# Terminal 2: Companion
wechat-claude
```

**Session Semantics Unification**
- Migrated from thread-specific abstraction to neutral session abstraction
- Codex, Claude, and shell adapters now share common session terminology
- Runtime session ID separated from persistent resume conversation ID
- Fixes "No conversation found" errors when resuming Claude conversations

**Local Companion Process**
New `local-companion.ts` module (242 lines):
- Unified companion process for both Codex and Claude
- Handles IPC protocol for companion communication
- Manages companion lifecycle and error recovery
- Supports three render modes: embedded, panel, companion

**Claude Hook Integration**
New `claude-hooks.ts` module (203 lines):
- Permission request forwarding to WeChat
- Session lifecycle event handling (`SessionStart`, `UserPromptSubmit`, `Stop`)
- Structured status updates instead of PTY scraping
- Remote confirmation support (`/confirm` and `/deny` commands)
- Automatic recovery from invalid resume errors

**Resume Compatibility Fixes**
- Transcript path parsing extracts resumable conversation ID
- Invalid resume detection with automatic fallback
- Clean separation of runtime and resume state
- Graceful error handling when resume fails

  const normalized = normalizeOutput(text);

## 2. Native Cross-Platform Support

### Background

Previously, the bridge was Windows-centric with hardcoded PowerShell assumptions. This release introduces first-class support for Linux and macOS while maintaining full backward compatibility with Windows.

### Implementation

**Platform-Aware Shell Selection**
- Windows: PowerShell (pwsh) or PowerShell Classic
- Linux: pwsh → bash → zsh → sh (fallback chain)
- macOS: pwsh → bash → zsh → sh (fallback chain)

**POSIX Shell Support**
Shell adapter now handles:
- bash (Bourne Again Shell)
- zsh (Z Shell)
- sh (POSIX shell)
- dash (Debian Almquist shell)
- ksh (KornShell)

**PTY Configuration**
- Windows: ConPTY (Console Pseudo Terminal)
- Linux/macOS: Unix PTY (pseudo-terminal)
- Proper TTY handling for each platform

**Environment Setup**
- Correct environment variable handling per platform
- PATH manipulation for shell detection
- Profile loading (.bashrc, .zshrc, etc.)

**Risk Command Detection**
Extended dangerous command detection for POSIX:
- Shell builtins (cd, exec, kill, etc.)
- Common destructive commands (rm, dd, mkfs, etc.)
- System modifiers (iptables, sysctl, etc.)
- Network operations (nc, nmap, curl, etc.)

## 3. Codex Busy State Fix

### Background

Previously, after Codex completed a reply, WeChat messages were incorrectly rejected with "codex is still working" due to a lingering in-memory active turn. This bug prevented users from sending follow-up messages without restarting the bridge.

### Implementation

**Root Cause**
- Active turn (`activeTurn`) not cleared after Codex reply completion
- Status remained "busy" instead of "idle"
- Next WeChat message incorrectly rejected

**Fix Strategy**
Ensure all completion paths clear state:
- Session `task_complete` event
- WebSocket `turn/completed` event
- Auto-completion scenarios
- Manual interruption

**Recovery Guard**
Added automatic cleanup for stale active turns:
- Detects when adapter state is idle but active turn persists
- Automatically purges stale turns
- Allows next WeChat message to proceed normally

**State Consistency**
- All code paths now clear `activeTurn` before emitting final stdout
- Status set to "idle" before message delivery
- Prevents future state inconsistencies

### Files Modified

- **`wechat-bridge.ts`** (127 lines modified)
  - Added state cleanup in all completion paths
  - Implemented recovery guard
  - Enhanced status management

- **`bridge-adapters.ts`** (modified)
  - Ensured all event handlers clear state
  - Added completion verification

### Impact

**Before**
```
[User] First message
[Codex] Completes reply
[User] Second message
[WeChat] ERROR: Codex is still working
[User] Must restart bridge
```

**After**
```
[User] First message
[Codex] Completes reply → State cleared
[User] Second message
[WeChat] Message accepted → Codex processes normally
```

## Breaking Changes

### Terminology Changes
- **Thread → Session**: Code and APIs now use session terminology
- `sharedThreadId` → `sharedSessionId` (deprecated but still works)
- `BridgeThread` → `BridgeSession` type

### Claude Entry Point
- Old experimental modes removed
- Use `wechat-claude` command for Claude companion

### Resume Behavior
- Claude resume now uses `resumeConversationId` instead of thread switching
- Resume failures auto-recover instead of crashing
