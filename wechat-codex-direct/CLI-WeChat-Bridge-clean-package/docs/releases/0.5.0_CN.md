# 版本 0.5.0

**发布日期：** 2026年3月24日

## 概述

版本 0.5.0 是一个变革性版本，从根本上重新构想桥接器架构。此版本引入完整的 Claude Code 原生支持，实现与 Codex 相同的双终端体验，统一所有适配器的会话语义，添加 Linux 和 macOS 的全面跨平台支持，并修复关键的状态管理问题——总计在 25 个文件中新增 3,595 行代码。

### 关键统计数据
- **25 个文件变更**
- **新增 3,595 行，删除 289 行**
- **3 个主要架构增强**
- **一流的跨平台支持**

## 1. Claude Code Companion 集成

### 背景

此前，Claude Code 集成是实验性的，缺乏 Codex 用户所享受的精细双终端体验。此版本引入完整的 Claude 原生支持，本地 companion 与 Codex 完全一致。

### 实现

**双终端架构**
Claude 现在支持与 Codex 相同的双终端工作流程：

```bash
# 终端 1：桥接器
wechat-bridge-claude

# 终端 2：Companion
wechat-claude
```

**会话语义统一**
- 从特定于线程的抽象泛化为中性会话抽象
- Codex、Claude 和 shell 适配器现在共享通用会话术语
- 运行时会话 ID 与持久化恢复会话 ID 分离
- 修复恢复 Claude 会话时的"No conversation found"错误

**本地 Companion 进程**
新的 `local-companion.ts` 模块（242 行）：
- Codex 和 Claude 的统一 companion 进程
- 处理 companion 通信的 IPC 协议
- 管理 companion 生命周期和错误恢复
- 支持三种渲染模式：embedded、panel、companion

**Claude Hook 集成**
新的 `claude-hooks.ts` 模块（203 行）：
- 权限请求转发到微信
- 会话生命周期事件处理（`SessionStart`、`UserPromptSubmit`、`Stop`）
- 结构化状态更新而非 PTY 抓取
- 远程确认支持（`/confirm` 和 `/deny` 命令）
- 从无效恢复错误自动恢复

**恢复兼容性修复**
- 转录路径解析提取可恢复的会话 ID
- 无效恢复检测与自动回退
- 运行时和恢复状态的清晰分离
- 恢复失败时的优雅错误处理

**权限请求流程**

5. 决策通过 hook 返回
6. Claude 继续操作

## 2. 原生跨平台支持

### 背景

此前，桥接器以 Windows 为中心，有硬编码的 PowerShell 假设。此版本在保持与 Windows 完全向后兼容的同时，引入了对 Linux 和 macOS 的一等支持。

### 实现

**平台感知 Shell 选择**
- Windows: PowerShell (pwsh) 或 PowerShell Classic
- Linux: pwsh → bash → zsh → sh（回退链）
- macOS: pwsh → bash → zsh → sh（回退链）

**POSIX Shell 支持**
Shell 适配器现在处理：
- bash (Bourne Again Shell)
- zsh (Z Shell)
- sh (POSIX shell)
- dash (Debian Almquist shell)
- ksh (KornShell)

**PTY 配置**
- Windows: ConPTY（控制台伪终端）
- Linux/macOS: Unix PTY（伪终端）
- 每个平台的正确 TTY 处理

**环境设置**
- 每个平台的正确环境变量处理
- 用于 shell 检测的 PATH 操作
- 配置文件加载（.bashrc、.zshrc 等）

**风险命令检测**
扩展 POSIX 的危险命令检测：
- Shell 内置命令（cd、exec、kill 等）
- 常见破坏性命令（rm、dd、mkfs 等）
- 系统修改器（iptables、sysctl 等）
- 网络操作（nc、nmap、curl 等）

## 3. Codex 忙碌状态修复

### 背景

此前，Codex 完成回复后，由于内存中残留活跃 turn，微信消息被错误拒绝为"codex is still working"。此错误阻止用户发送后续消息，需要重启桥接器。

### 实现

**根本原因**
- Codex 回复完成后未清除活跃 turn（`activeTurn`）
- 状态保持"busy"而不是"idle"
- 下一条微信消息被错误拒绝

**修复策略**
确保所有完成路径清除状态：
- 会话 `task_complete` 事件
- WebSocket `turn/completed` 事件
- 自动完成场景
- 手动中断

**恢复守卫**
为残留活跃 turn 添加自动清理：
- 检测适配器状态为空闲但活跃 turn 持续存在
- 自动清除残留 turn
- 允许下一条微信消息正常进行

**状态一致性**
- 所有代码路径现在在发出最终 stdout 之前清除 `activeTurn`
- 在消息传递之前将状态设置为"idle"
- 防止未来的状态不一致

### 修改文件

- **`wechat-bridge.ts`**（修改 127 行）
  - 在所有完成路径中添加状态清理
  - 实现恢复守卫
  - 增强状态管理

- **`bridge-adapters.ts`**（修改）
  - 确保所有事件处理程序清除状态
  - 添加完成验证

### 影响

**之前**
```
[用户] 第一条消息
[Codex] 完成回复
[用户] 第二条消息
[微信] 错误：Codex 仍在工作中
[用户] 必须重启桥接器
```

**之后**
```
[用户] 第一条消息
[Codex] 完成回复 → 状态清除
[用户] 第二条消息
[微信] 消息被接受 → Codex 正常处理
```

## 重大变更

### 术语变更
- **线程 → 会话**：代码和 API 现在使用会话术语
- `sharedThreadId` → `sharedSessionId`（已弃用但仍有效）
- `BridgeThread` → `BridgeSession` 类型

### Claude 入口点
- 删除旧的实验模式
- 使用 `wechat-claude` 命令启动 Claude companion

### 恢复行为
- Claude 恢复现在使用 `resumeConversationId` 而不是线程切换
- 恢复失败自动恢复而不是崩溃
