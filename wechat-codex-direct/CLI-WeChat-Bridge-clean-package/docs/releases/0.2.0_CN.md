# CLI WeChat Bridge 发布说明 - v0.2.0

## 概述

**v0.2.0** 代表了 Codex 微信桥接的完整重新设计，引入了双终端架构，在保留原生 Codex TUI 体验的同时，通过微信提供清晰的结构化最终回复。此版本从根本上改变了桥接的工作方式，从在伪终端中嵌入 Codex 转变为运行原生 Codex 面板和基于结构化 websocket 的远程控制接口。

这是桥接功能的第一个稳定版本，为所有未来开发奠定了基础。

## 主要功能

### 原生 Codex 终端体验

- **原生 TUI 面板**：Codex 现在在真正的原生终端中运行，完全支持 TUI，而不是嵌入式 PTY 模拟
- **可见面板**：用户可以看到真实的 Codex 界面，所有原始功能完整保留——alt-screen、光标导航和交互控件按预期工作
- **无文本注入退化**：终端由 Codex 本身直接驱动，消除了逐字符注入的伪影和显示损坏

### 通过微信进行结构化远程控制

- **清晰的最终回复**：微信仅接收最终答案文本，而非终端片段、TUI 重绘或内部状态噪声
- **结构化轮次管理**：远程消息通过 Codex 的 app-server websocket API 发送，而非 PTY 按键注入
- **任务完成检测**：桥接器准确识别 Codex 何时完成任务并发送单一、连贯的回复
- **无协议污染**：中间信号如 `task_complete`、持续时间、心跳消息和状态栏文本从微信聊天流中过滤掉

### 简化的授权模型

- **移除 `/pair` 机制**：消除了临时配对工作流程，转而使用直接凭据绑定
- **单所有者模型**：`account.json.userId` 中的微信账号是唯一授权用户
- **无运行时重新绑定**：所有者身份在设置期间建立一次，在运行期间永不改变

### 改进的 Windows 兼容性

- **PowerShell 脚本避免**：自动检测并绕过 `codex.ps1` 以避免 PowerShell 执行策略问题
- **供应商 exe 优先级**：优先使用供应商目录中的官方 `codex.exe` 而非临时副本
- **健壮的路径解析**：使用正确的命令行转义处理 `.cmd` 和 `.exe` 入口点

## 架构变更

### 双进程设计

桥接器现在由两个协调的进程组成：

1. **桥接进程**（`wechat-bridge-codex`）：管理微信连接并维护结构化轮次状态
2. **面板进程**（`wechat-codex`）：运行具有完整终端功能的原生 Codex TUI

这种分离实现了：
- 真正的原生终端行为
- 无需 PTY 技巧的结构化远程控制
- 关注点清晰分离，更易于调试

### Codex App-Server 集成

- 与 Codex 的 app-server 建立直接 websocket 连接以进行结构化操作
- 支持 `thread/start`、`turn/start`、`turn/interrupt` RPC 调用
- 处理命令执行和文件更改的服务器端审批
- 正确的 `experimentalApi` 能力协商

## 重要说明

### 与预发行版本的重大变更

如果您在 v0.2.0 之前使用的是实验版本：

- **不再有 `/pair` 命令**：删除所有保存的配对状态；授权现在直接使用 `account.json`
- **面板入口点已更改**：使用 `wechat-codex` 而非嵌入式模式
- **线程状态迁移**：共享线程现在在桥接重启之间持久化和恢复
- **PTY 伪影已消除**：任何依赖解析 PTY 输出的工作流程都需要调整

### 迁移操作

1. 确保您的微信凭据正确保存在 `~/.claude/channels/wechat/account.json` 中
2. 重新启动桥接器和面板以建立新架构
3. 验证您账户的 `userId` 被识别为授权所有者
4. 测试原生 Codex 面板交互——键盘输入和显示应完全像独立 Codex 一样工作

## 升级说明

从 v0.2.0 之前的版本升级：

```bash
# 拉取最新代码
cd CLI-WeChat-Bridge
git pull

# 安装依赖
bun install

# 重新安装全局命令
npm install -g .

# 在项目目录中启动桥接器
cd D:\work\your-project
wechat-bridge-codex

# 在第二个终端中，启动 Codex 面板
wechat-codex
```
